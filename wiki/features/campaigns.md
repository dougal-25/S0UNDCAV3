# Feature — Campaigns (Phase 3)

> Status: **Shipped v0 2026-05-13.** Authoritative spec: [`phase_2_3_pivot.md`](../spec/phase_2_3_pivot.md). Mission brief: `~/Downloads/Soundcave Phase 2.3 Mission.md`.

## What it is

A **Campaign** is the sequenced bundle of posts auto-generated for a single Event. One campaign per event (v0). Each campaign holds an ordered timeline of `posts` — announcement, headliner spotlight, support spotlights, mid-push, countdowns, day-of, recap — with copy variants generated by Claude using the Event's voice preset.

## Why it exists

The Phase 2/3 wedge: "turn one event into a month of content." Campaigns are the content. Without this, [Events](events.md) are just a calendar. With this, every event becomes ~10–16 ready-to-edit posts a promoter would otherwise spend half a day writing.

## How it works (v0)

1. Promoter opens an Event detail page. The campaign section shows an empty-state with `{GENERATE CAMPAIGN}` if no campaign yet exists.
2. Click triggers `POST /api/events/<id>/generate-campaign`. The server:
   - Fetches Event + lineup + linked artist Profiles.
   - Runs the **template engine** (`campaign_template.build_campaign`) to compute the post sequence based on the event date, lineup size, and how far out the event is.
   - Creates a `campaigns` row (`status: generating`).
   - For each planned post, calls Claude with the **voice preset's system prompt** + the **post-type brief** + the **Event/Profile data** as the user message. **Haiku** for bulk posts, **Sonnet 4.6** for hero post types (`announcement`, `headliner_spotlight`, `recap`).
   - Inserts each `posts` row with 3 copy variants (`copy_variants` JSONB).
   - Marks campaign `ready` when done; surfaces `generation_error` on partial failure.
3. UI re-loads the detail view and renders the timeline.

**v0 is synchronous.** 6–14 posts × ~1–2s each = ~15–30s. Acceptable for one user at a time. Phase 3 v1 moves this to background threads + polling.

## Template engine

`campaign_template.build_campaign(event_date, lineup)` is a pure function that returns the post sequence. Logic:

- **Announcement:** earliest possible, capped at 42 days out. Skipped if the event is sooner than 1 day from now.
- **Spotlights:** one per lineup artist (headliner first), staggered evenly between (announcement + 3 days) and (event − 10 days). Past dates skipped.
- **Mid-campaign push:** 14 days out (if in the future).
- **Countdown sequence:** 7d, 3d, 1d, day-of (all scheduled at event time on the respective day).
- **Day of doors:** event day, 1 hour before event start (clamped ≥18:00).
- **Recap:** 24 hours after.

Posts are sorted chronologically before insert.

## Voice presets

`config/voice_presets.py` exports 4 presets — `underground`, `professional`, `high_energy`, `intimate` — each with:
- A system-prompt fragment (tone, vocabulary, sentence length, emoji policy).
- An `image_mood` (for future Phase 3 v1 image-bg generation).

`system_prompt_for(voice)` concatenates the preset + **global hard rules** that apply across all voices:
1. No AI-clichés (`dive into`, `unlock`, `immerse`, etc.)
2. **Never invent facts** — no fictional set times, opening DJs, prices, capacity.
3. **Never print numeric stats** — follower counts, play counts, etc. Use them to inform tone, not as copy facts.
4. No generic openers (`Join us`, `Get ready`).
5. No vague venue/scene worship (`changes everything about what's possible on a dancefloor`).
6. No hashtags unless asked.
7. Respect platform character limits.

## Post-type briefs

`POST_TYPE_BRIEFS` in `campaigns_api.py` — one string per `post_type` enum value. Tells Claude what the post is FOR (announcement = reveal, countdown_3d = urgency, recap = warm reflection). Tunable file.

## Data model

| Table | Key columns |
|---|---|
| `campaigns` | `event_id` (unique), `status` enum (`generating / ready / partially_published / completed / archived`), `voice_preset` (denormalised), `generation_started_at`, `generation_completed_at`, `generation_error` |
| `posts` | `campaign_id` (cascade), `post_type` enum, `scheduled_for`, `target_platforms` text[], `linked_artist_profile_id`, `copy_variants` JSONB, `selected_copy_variant_id`, `image_asset_urls` text[], `regeneration_count`, `publish_status` enum, `generation_error` |

RLS: scoped via parent Event ownership for both tables.

Migration: `db/0014_campaigns_posts.sql`.

## API surface

| Verb | Path | Purpose |
|---|---|---|
| POST | `/api/events/<id>/generate-campaign` | Plan + generate. `?regenerate=true` to wipe and rebuild. |
| GET | `/api/campaigns/<id>` | Campaign with nested posts. |
| GET | `/api/events/<id>/campaign` | Convenience — returns `{campaign, posts}` for an event. |
| POST | `/api/posts/<id>/regenerate-copy` | Re-run copy gen for a single post. |
| PATCH | `/api/posts/<id>` | Select variant / edit copy / reschedule / change platforms. |

All JWT-gated, ownership enforced via event walkthrough.

## Output parsing

Claude returns variants as `{variants: [{id, text}, ...]}` — but in practice it sometimes wraps in ```json``` fences, appends "FLAGGING SOMETHING…" prose, or uppercases keys. `_extract_first_json_object` walks the response char-by-char tracking string literals and brace depth to extract the first balanced JSON block. `_parse_variants_json` tolerates uppercase keys and falls back to one clean text variant if parsing fails — the UI never sees raw JSON.

## What's NOT here (v1+)

- **Image composition** — Phase 3 v1. Pillow pipeline composing real Profile photos + AI-generated abstract backgrounds + typography per post-type layout template.
- **Async generation** — background threads + Supabase realtime / polling for progressive timeline population. Current sync flow is fine for single-user beta.
- **Post-editor modal** — variant picker, free-text edit, per-post regenerate UI. Currently only the first variant renders; the others sit in `copy_variants` JSONB.
- **Bulk actions** — multi-select + bulk schedule/skip/regenerate.
- **Multi-campaign per event** — useful for "ticket release wave 1" vs "wave 2" pushes; deferred.
- **Voice training from real event copy** — Doug's north-star ask 2026-05-13; explicitly post-beta per mission brief.

## Known issues

- Copy quality is preset-dependent and the presets are hand-tuned. The brief is honest about this: tuning these is the single biggest lever on output quality, and richer artist data (Phase 4 claim flow) will help spotlights significantly.
- `js/events.js` is still over 500 lines (debt task #23). The campaign timeline UI lives there; splitting `events_detail.js` out should happen before more UI work.
